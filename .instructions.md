# Contributing to redactor -- Git Topology Guide

## Canonical source of truth

| | |
|---|---|
| **GitHub** | https://github.com/wwwizards/redactor |
| **Branch** | `main` |
| **PyPI** | not yet published (`pip install -e .` for local use) |

All changes to this module should be committed and pushed to the repo above. That is the authoritative history regardless of how you obtained these files.

## Step 1 -- Determine your local git context before touching anything

This directory contains a `.git/` folder. That means it is its own git repository. However, depending on how you obtained these files, you may be working inside a **parent repository** that also tracks this directory -- and that changes how commits work.

Run this from inside the `redact/` directory:

```bash
git remote -v
git log --oneline -5
```

Then run this from the directory **above** `redact/` (your parent project, if any):

```bash
git remote -v
git status
```

Use the table below to identify your situation:

| What you see | Situation | What to do |
|---|---|---|
| `redact/` has a remote pointing at `wwwizards/redactor` | **Standalone clone** (normal) | Commit and push inside `redact/` as usual |
| Parent dir has a remote; `git status` there shows `redact/` as a **submodule** (line starts with `modified: redact (new commits)`) | **Registered submodule** | Commit inside `redact/`, push to `wwwizards/redactor`, then bump the submodule pointer in the parent |
| Parent dir has a remote; `git status` there shows files inside `redact/` as **plain modified files** | **Unregistered nested repo** (dual-context hazard) | See below |

## The dual-context hazard (unregistered nested repo)

If a parent repository tracks files inside `redact/` as plain blobs (not a submodule), both git histories advance independently with no link between them.

Signs you are in this situation:
- `git status` in the parent shows `modified: redact/redactor.py` (a file path, not `modified: redact`)
- Running `git commit` in the parent produces a commit, but `git log` inside `redact/` is unchanged
- `git log` inside `redact/` still shows `Initial commit` after you "committed" from the parent

**This is the silent failure mode.** The parent commit looks complete but `wwwizards/redactor` never received the changes. Anyone cloning from GitHub gets stale code.

To avoid polluting either history:

```bash
# Always commit to the canonical upstream first
cd path/to/redact
git add <changed files>
git commit -m "vX.Y.Z - description"
git push origin main

# Then record the bump in the parent if needed
cd ..
git add redact/
git commit -m "bump redact to vX.Y.Z"
```

## Step 2 -- Before starting any session, verify upstream is current

```bash
cd path/to/redact
git fetch origin
git log --oneline origin/main -5   # what does GitHub actually have?
git status                          # any local changes not yet committed?
```

If your local HEAD is behind `origin/main`, pull first. If it is ahead, push before doing other work. Never start a new feature on top of an unsynced base.

## Step 3 -- Run the tests before and after any change

```bash
pytest test_redactor.py -v
```

36 tests, ~6 seconds, zero network calls. See [TESTING.md](TESTING.md) for the full matrix. All tests must pass before pushing.

## If you find this directory inside a larger monorepo

The cleanest long-term fix is to register `redact/` as a proper git submodule in the parent:

```bash
# From the parent repo root (remove the plain directory first if needed)
git submodule add https://github.com/wwwizards/redactor.git path/to/redact
git commit -m "register redactor as submodule"
```

This causes `git status` in the parent to show a submodule pointer diff instead of silently tracking file blobs, eliminating the dual-context hazard entirely.

## Tools that may help

**[pickaxe](https://github.com/wwwizards/pickaxe)** is a repo-health and discovery toolkit that may be useful if you are managing a monorepo with multiple nested or vendored projects. Among other things it can scan for nested `.git/` directories and unregistered submodules -- exactly the topology that creates the dual-context hazard described above. If you find yourself managing several projects like this one inside a larger workspace, pickaxe may save you from rediscovering this problem the hard way.
